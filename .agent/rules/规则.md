### 编程规则：全链路中文工作流

1. 核心语言要求

所有的交互、推理、计划和文档必须完全使用 简体中文。

除非是代码中的特定变量、库名或通用的行业术语（如 Props, Component, API 等），否则严禁使用英文。

2. 交付物（Artifacts）标准

Task Lists (任务列表)：每一项任务的描述必须是中文，清晰指明操作目标。

Implementation Plans (执行方案)：架构设计、逻辑流程、步骤拆解必须使用中文撰写，确保方案易于理解。

Walkthroughs (代码演示)：在完成修改后，必须用中文总结变更点、测试步骤和验证结果。

3. 代码注释与文档

所有的代码逻辑保持标准编程语法。

核心要求：所有新生成的或修改的 代码注释 (Comments)、JSDoc/Docstrings 以及项目文档必须使用中文。

4. 术语处理

遇到关键技术术语，首选“中文名称 (英文名称)”的格式，例如：“中间件 (Middleware)”。

5. 技术栈与架构规范

- **Next.js 版本**：项目固定使用 **Next.js 16**。
- **异步 API 处理**：必须遵循 Next.js 16 的规范，所有 `params`、`searchParams`、`cookies()`、`headers()` 等 API 必须作为 **Promise** 处理，使用 `await` 或 React 的 `use()` 进行解构。严禁同步访问这些属性。

6. 称呼规范

- **用户称呼**：每次与用户交流时，必须称呼用户为"**相公**"。无论是回答问题、汇报进度还是请求确认，均须在开头或行文中使用此称呼。

7. 数据库迁移规范

- **强制使用 `db:migrate`**：所有数据库 Schema 变更必须通过迁移文件管理，严禁在日常开发中使用 `db:push`。
- **标准工作流**：
  1. 修改 `src/shared/api/schema/` 中对应的 Schema 文件
  2. 运行 `pnpm db:generate` 生成迁移 SQL 文件
  3. 检查 `drizzle/` 下新增的 `.sql` 文件是否符合预期
  4. 运行 `pnpm db:migrate` 应用迁移到数据库
  5. 将 `drizzle/` 下的迁移文件纳入 Git 版本控制
- **禁止事项（铁律）**：
  - ❌ 禁止使用 `pnpm db:push` 同步 Schema，该命令不生成迁移记录，会导致环境间不一致和部署冲突
  - ❌ **严禁** 通过 Navicat、DataGrip、pgAdmin 或任何数据库 GUI 工具在线上数据库中**手动新建/删除表、添加/删除列**
  - ❌ **严禁** 直接在 ECS 服务器上连接数据库后执行 DDL 语句（`CREATE TABLE`、`DROP TABLE`、`ALTER TABLE`）
  - 违反以上规定会导致数据库实际状态与 Git 中的迁移脚本对不上，后续所有迁移将全部卡死（见 2026-03 生产事故）
- **正确的删表流程**：
  1. 在 `src/shared/api/schema/` 中找到对应表的 `pgTable()` 定义，将整个定义块**删除**
  2. 运行 `pnpm db:generate`，Drizzle 会自动生成包含 `DROP TABLE` 的迁移 SQL
  3. 检查生成的 `.sql` 文件内容是否正确
  4. 将迁移文件提交到 Git 后，通过 `pnpm db:migrate` 或部署流程应用到所有环境
- **Enum 数据规范**：
  - 在编写迁移脚本前，若涉及 Enum 类型字段（如 `role`），必须先确认数据库中**不存在非法的历史数据**
  - 可通过以下查询检查：`SELECT DISTINCT role FROM users WHERE role NOT IN ('ADMIN', 'SUPER_ADMIN', ...);`
