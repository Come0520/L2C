## 修复日志服务字段命名规范问题

### 问题分析
1. **日志服务存在字段命名不匹配**：
   - 创建日志时使用前端驼峰命名直接插入数据库（如 `ipAddress`、`userAgent`、`createdAt`）
   - 查询时却使用正确的蛇形命名（如 `created_at`）
   - 违反项目“数据库表名和字段名必须使用 snake_case”的强制规则

2. **缺乏通用的字段转换工具**：
   - 项目中已有 `toCamelCase` 函数，但缺少 `toSnakeCase` 函数
   - 多处服务使用手动映射，缺乏统一的转换层

### 修复方案

#### 1. 添加通用的蛇形命名转换函数
- 在 `src/utils/string.ts` 中添加 `toSnakeCase` 函数
- 实现驼峰命名到蛇形命名的转换

#### 2. 修复日志服务的字段映射
- 修改 `src/services/logs.client.ts` 中的 `createLog` 方法
- 添加前端驼峰命名到数据库蛇形命名的转换逻辑
- 确保所有数据库操作使用正确的蛇形命名字段

#### 3. 统一转换策略
- 遵循 `products.client.ts` 的转换模式
- 实现明确的字段映射，确保前后端字段一致性

### 修复步骤

1. **添加 `toSnakeCase` 函数**：
   ```typescript
   // 在 string.ts 中添加
   export function toSnakeCase(str: string): string {
     return str.replace(/([A-Z])/g, '_$1').toLowerCase();
   }
   ```

2. **修改日志服务的字段转换**：
   ```typescript
   // 在 createLog 方法中添加转换逻辑
   const dbData = {
     id: newLog.id,
     user_id: newLog.userId,
     user_name: newLog.userName,
     action: newLog.action,
     level: newLog.level,
     resource_id: newLog.resourceId,
     resource_type: newLog.resourceType,
     details: newLog.details,
     ip_address: newLog.ipAddress,
     user_agent: newLog.userAgent,
     created_at: newLog.createdAt
   };
   ```

3. **更新日志服务的查询逻辑**：
   - 确保所有查询条件使用正确的蛇形命名字段

### 预期结果
- 日志服务的所有数据库操作都使用正确的蛇形命名字段
- 解决插入失败或运行时不一致问题
- 遵循项目的命名规范强制要求
- 为其他服务提供统一的字段转换参考
