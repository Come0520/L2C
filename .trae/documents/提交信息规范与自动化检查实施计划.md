# 提交信息规范与自动化检查验证计划

## 项目现状确认
- ✅ 已安装 `@commitlint/cli` 和 `@commitlint/config-conventional` 依赖
- ✅ 已配置 `commitlint.config.js`，使用 Conventional Commits 规则
- ✅ 已安装 `husky` 依赖并初始化
- ✅ 已配置 `.husky/commit-msg` 钩子，内容为 `npx --no-install commitlint --edit "$1"`
- ✅ 已配置 `.husky/pre-commit` 钩子，包含 TypeScript 检查、ESLint 检查和测试
- ✅ 已配置 `lint-staged`

## 验证步骤

### 1. 测试拦截（Negative Test）
- **目标**：验证不规范的提交信息被正确拦截
- **测试命令**：`git commit -m "test commit"`（不符合 Conventional Commits 规范）
- **预期结果**：提交被拦截，显示 commitlint 错误信息
- **成功标志**：git 提交失败，输出类似 "type must be one of [feat, fix, docs, style, refactor, test, chore, perf, ci, build, revert]"

### 2. 测试通过（Positive Test）
- **目标**：验证规范的提交信息能正常提交
- **测试命令**：`git commit -m "chore: test commit linting"`（符合 Conventional Commits 规范）
- **预期结果**：提交成功，或进入 pre-commit 钩子的其他检查
- **成功标志**：git 提交过程开始执行 pre-commit 钩子的其他检查（TypeScript、ESLint、测试等）

### 3. 文档检查与更新
- **目标**：确保项目文档包含提交信息规范说明
- **检查内容**：查看 `README.md` 或 `CONTRIBUTING.md` 文件
- **更新内容**：如果缺少，添加 Conventional Commits 规范说明和示例
- **成功标志**：文档中包含提交信息格式、type 说明和示例

## 预期结果
- 确认现有的 commitlint 和 husky 配置工作正常
- 开发人员必须遵循 Conventional Commits 规范提交代码
- 项目文档包含清晰的提交信息规范说明

## 提交信息规范示例
```
# 特性添加
feat(user): add login functionality

# 修复bug
fix(auth): resolve login failure issue

# 文档更新
docs: update README with commit guidelines

# 代码重构
refactor(user): simplify user profile component

# 测试添加
test(auth): add login test cases

# 构建相关
chore(build): update dependencies
```

## 验收标准
- ✅ 不规范的提交信息被 git 钩子拦截
- ✅ 规范的提交信息能正常进入后续检查
- ✅ 项目文档包含提交信息规范说明（如缺失则添加）
