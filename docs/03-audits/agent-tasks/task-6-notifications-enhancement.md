# Task 6: Notifications(通知)模块 L4 升级指南

> 专属 Subagent：负责 `src/features/notifications` 目录

## 背景

当前 notifications 模块成熟度为 L3 (6.1)，不仅缺失测试，更主要的是用户交互缺乏分类筛选机制，难以进行复杂流的排查（无日志接入）。需要进行这三方面的统一补强。

## 严格工作范围

- **仅限修改**：`src/features/notifications/`
- **绝对禁止修改**：任何其他业务模块、`src/shared/`、后台入口配置等。

## 详细任务

### 1. UI 升级与体验重构（D5 提升）
- 定位通知中心组件。目前的列表可能只是一个单一混排列表。
- 增加按通知类型（如系统、审批、业务提醒）的 Tab 分类，或者添加筛选操作区。增加“全部已读”按钮的安全防护实现。

### 2. 补足自动化测试（D3 提升）
- 编写测试文件验证：向特定用户推送多条通知 -> 用户执行全部标记为已读 -> 未读 count 是否正确。
- 包括分类统计逻辑的测试校验。

### 3. 审计与可运维性接入（D7 提升）
- 确保在群发全员通知、重要的强制通知流转等关键 Actions 中接入全局的 `AuditService`。
- 添加必要的 `logger.info` 帮助线上排错。

### 4. 需求文档更新（D4 提升）
- 编写/更新 `docs/02-requirements/modules/通知.md` （如果有该文件或者属于 `辅助功能.md`）。写明现有的触发规则与类型结构。

## 完工自检清单

- [ ] 完成分类展示重构并跑通原有功能。
- [ ] 全部已读等流转逻辑测试 100% 成功（`npx vitest run src/features/notifications`）。
- [ ] 对关键通知写入操作埋点了审计及标准 LOG。
- [ ] 类型检查通过：`npx tsc --noEmit` 未引起故障。
