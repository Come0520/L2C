# Task 7: Dashboard Module Refactoring & Enhancement (L3 → L4)

> **目标**：全面清剿 Dashboard 模块的历史技术债（约 30 处 TODO 集中区），优化首屏渲染性能，将其成熟度提升至 L4 安全生产就绪标准。

## 背景与痛点
根据成熟度审查报告，当前 `dashboard` 模块是全项目 `// TODO:` 注释最集中的区域。它作为用户登录后的首个到达面，直接决定了系统第一印象。其主要的缺失集中在：
- 遗留的 Mock 数据或未闭环的临时逻辑。
- 缺少关键性能层面的组件级懒加载和数据预取。
- Loading/Error 三态处理不完整。
- 作为核心数据汇总出口，缺失针对关键调用的审计日志（AuditService）和埋点（logger）。

## 工作目录范围
你**只能**修改或添加以下路径下的文件。禁止越权修改其他模块或共享组件，但可以引用它们：
- `src/features/dashboard/` 及其所有子目录

## 核心任务与验收标准

### 任务 1：清剿 TODO 与临时代码 (核心目标)
- **要求**：全局搜索 `src/features/dashboard` 目录下的 `TODO` 和 `FIXME`。阅读它们所在的上下文逻辑，将它们逐一实现并闭环。
- **验证**：整个 `dashboard` 目录内的 TODO 注释必须降至 5 个以内（最好清零）。对于确实无法在前端单方面解决的底层基建 TODO，请保留并补充详细说明。

### 任务 2：首屏性能优化与数据预取
- **要求**：根据 React 最佳实践，排查瀑布流请求（Waterfalls）。对于不依赖首屏阻塞的次要数据面板组件（如历史动态、非核心图表），引入 `React.lazy` 或类似机制进行懒加载。
- **验证**：对于需要 SSR 加载的数据页，在 actions 或 services 层配置合理的缓存参数（如适用）。首屏关键面板数据应该并行发起请求而非串行。

### 任务 3：UI 三态（Loading/Error/Empty）兜底
- **要求**：由于涉及到多个独立的看板和 Widget 组件，必须确保每个组件在数据拉取阶段有优雅的 Skeleton 骨架屏过度，且遇到异常（如超时）时有局部友好的错误边界（Error Boundary），不可因单个面板崩溃而导致整个 Dashboard 白屏。
- **验证**：至少增加或完善两种以上的 Skeleton 状态。

### 任务 4：引入审计与日志监控 (AuditService & Logger)
- **要求**：排查 dashboard actions 中的核心查询或用户关注度极高的数据看板（例如大盘统计刷新），合理加入 `AuditService.log` 进行审计留存，同时对可能的异常通过 `logger.error` 回传。
- **验证**：至少在 dashboard 层有 2 处以上的规范日志上报。

### 任务 5：TS 类型清理与单元测试补充
- **要求**：如果存在任何 `any` 类型，必须定义具体的 Zod Schema 或 TS 接口消除它。
- **验证**：`tsc --noEmit` 保证没有任何该目录相关报错。根据现有的图表或面板组件补充部分基础的测试（若无测试基建可酌情补充关键纯函数的单元测试）。

3. 请大声宣告“Task 7 完成”，详细汇报清除了哪几处关键 TODO。等待主线程的全量复核。
