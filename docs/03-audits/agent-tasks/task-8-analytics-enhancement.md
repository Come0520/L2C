# Task 8: Analytics Module Enhancement (L3 → L4)

> **目标**：强化数据分析模块（Analytics）在报表查询层面的性能与准确性，增加安全与可运维支撑，稳步从 L3 跃升至 L4 级别。

## 背景与痛点
Analytics（数据分析室）负责呈现企业经营核心链路的各类切片数据（销售漏斗、效能图表等）。根据系统扫描，该模块虽然功能大体可用，但在报表查询的索引配置优化、渲染性能防抖，特别是关键数据的安全审计与防透传上尚存缺陷。若不做兜底容易在大数据量时成为性能瓶颈或存在敏感信息查询无痕的隐患。

## 工作目录范围
你**只能**修改或添加以下路径下的文件：
- `src/features/analytics/` 及其所有子目录

## 核心任务与验收标准

### 任务 1：报表查询性能与索引优化
- **要求**：审查当前的 `analytics/services` 或 actions 中调用的长周期查询（例如跨表 Join 的复杂统计）。排查是否有存在潜在的 N+1 问题，或是否可以直接在 ORM 端通过更优的聚合查询（groupBy/count 等）平替循环汇总。
- **验证**：对耗时较长、复杂度较高的数据操作逻辑尝试拆分或应用 `unstable_cache`（Next.js）进行缓存封装（如果适用）。

### 任务 2：针对敏感导出/查询接入 AuditService
- **要求**：数据报表可能包含公司的核心财务或业绩机密。必须在触及“大范围数据统计查询”或“报表文件生成导出”（如果有）时接入 `AuditService`。
- **验证**：在对应的 `actions` 中引用并执行 `AuditService.log(db, { ... })` ，规范记录执行人和查询类型。

### 任务 3：图表渲染体验加固 (Skeleton & Error Boundary)
- **要求**：为包含复杂图形的组件提供独立的错误捕获容器。提供类似 Dashboard 要求的优雅 Skeleton 并提升 FCP 体验。
- **验证**：至少新增 1 处 ErrorBoundary 的运用以及对应的降级 UI 提示。

### 任务 4：补齐单元测试 / 集成测试
- **要求**：针对报表数据的日期转换、金额汇总计算等核心纯函数补全单元测试。若没有，请在 `__tests__` 目录下创建以覆盖至少 3 个图表或数据流向的核心逻辑分支。
- **验证**：测试覆盖通过（包含正逆双向）。

## 交付说明
在你认为一切就绪后：
1. 请运行 `npx tsc --noEmit` 并确认无任何 TS 类型回退。
2. 运行 `npx vitest run src/features/analytics` 确认相关业务测试覆盖 100% 通过。
3. 请宣告“Task 8 完成”，附上性能优化与安全修补点清单。等待主线程的全量复核。
