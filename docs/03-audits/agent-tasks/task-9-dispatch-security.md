# Task 9: Dispatch Module Security Patch (L2 → L3)

> **目标**：紧急封堵调度引擎（Dispatch）在多租户系统下的硬伤死穴（缺失 tenantId 硬隔离），作为 L2 级的核心功能，安全基建必须首轮整改，确保越过 L3 合格分数线。

## 背景与痛点
根据 `module-maturity-status.md` 的警告，整个项目中绝大部分模块都已严格执行了多租户（Multi-tenant）的硬隔离策略，唯有 `dispatch`（服务调度与派单）**缺失了针对 `tenantId` 的数据隔离限制**。在 SaaS 系统中，越权查改属于 P0 级严重缺陷（串场风险）。必须将其优先根除，并补齐基本的测试保证基础流转无误。

## 工作目录范围
你**只能**修改或添加以下路径下的文件：
- `src/features/dispatch/` 及其所有子目录

## 核心任务与验收标准

### 任务 1：全面补齐 `tenantId` 硬隔离隔离（P0级）
- **要求**：检索 Dispatch 模块内所有的 Database Queries 和 Actions 操作。无论创建调度单、指派人员、流转状态或分页查询，都**必须无条件携带并验证 `tenantId`** 过滤。
- **验证**：任何 `.where(...)` 的 SQL 组合在涉及租户专属数据表中，`eq(table.tenantId, tenantId)` 必须不可或缺。

### 任务 2：增加安全拦截边界的集成测试
- **要求**：在 `__tests__` 下为主干行为构建基本的生命周期集成用例，并**刻意制造跨租户修改企图**，以验证系统的隔离性。
- **验证**：至少覆盖 3 个测试用例。要求在尝试查询异租户数据时抛出异常或返回空结果。

### 任务 3：消灭历史 any 类型与梳理基础说明文档
- **要求**：在加入 `tenantId` 的同时，顺手审查是否有 `any` 并替之为具体 Schema 定义。并在 `docs` 下生成一份基于当前调度的简明 API 机制文档（包含流转状态的预定义）。
- **验证**：该模块非测试用例中的所有 ts 源码必须做到 `any`/`as any` 清零。

## 交付说明
在你认为一切就绪后：
1. 请运行 `npx tsc --noEmit` 检查语法及泛型变更的一致性无误。
2. 运行 `npx vitest run src/features/dispatch`，确保新增的隔离拦截断言被激活并通过。
3. 宣告“Task 9 完成”，汇报具体修复了哪些缺失 `tenantId` 的高危接口。等待主线程的全量复核。
