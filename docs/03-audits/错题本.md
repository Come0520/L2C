## D2 - 代码质量类

### D2-001 Next.js 16 异步 API 同步访问（🔴 Critical）

- **问题描述**：在 Next.js 16 中，`params`、`searchParams`、`cookies()`、`headers()` 均为 Promise 类型，但开发者习惯性地将其作为同步对象访问。
- **典型反例**：
  ```tsx
  export default function Page({ params }: { params: { id: string } }) {
    const id = params.id; // 🚨 编译通过但运行时错误
  }
  ```
- **正确写法**：
  ```tsx
  export default async function Page({ params }: { params: Promise<{ id: string }> }) {
    const { id } = await params;
  }
  ```
- **快速检测**：`grep -rn "params\." src/app --include="*.tsx" | grep -v "await params"`
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：-

### D2-002 useActionState 误用为 useFormState（🟠 High）

- **问题描述**：React 18 的 `useFormState` 已在 React 19 中更名为 `useActionState`，混用会导致类型错误和 deprecation 警告。

  ```tsx
  import { useFormState } from 'react-dom'; // 🚨 已废弃
  ```

- **正确写法**：

### D2-003 Client Component 提升过高（🟡 Medium）

- **问题描述**：本可作为 Server Component 的父容器，因为内部一个小交互组件而整体加了 `'use client'`，导致整棵子树都变为客户端渲染，丧失 RSC 优势。
- **典型反例**：

## D3 - 安全类

### D3-001 Server Action 权限锁缺失（🔴 Critical）

- **问题描述**：Server Action 函数体内没有在**第一行**调用 `await auth()` 获取会话并用 `checkPermission(session, 'xxx.xxx')` 校验权限码，导致未登录或低权限用户可以直接调用该 Action。
- **典型反例**：
  ```ts
  'use server';
  export async function deleteOrder(id: string) {
    await db.delete(ordersTable).where(eq(ordersTable.id, id)); // 🚨 无鉴权
  }
  ```
- **正确写法**：

  ```ts
  'use server';
  import { auth, checkPermission } from '@/shared/lib/auth';
  // 权限码参见: src/shared/config/permissions.ts

  export async function deleteOrder(id: string) {
    // Step 1: 获取会话（第一行铁律）
    const session = await auth();
    if (!session?.user) return { success: false, error: '未登录' };
    // Step 2: 校验权限码（而非角色名）
    const ok = await checkPermission(session, 'order.delete');
    if (!ok) return { success: false, error: '无权限' };
    // Step 3: 业务逻辑
    await db
      .delete(ordersTable)
      .where(and(eq(ordersTable.id, id), eq(ordersTable.tenantId, session.user.tenantId)));
  }
  ```

- **快速检测**：对比 `features/*/actions/*.ts` 下所有 `export async function` 的第一行是否为 `await auth()`
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：quotes (2026-03-10), orders (2026-03-10)

### D3-002 多租户隔离缺失（tenantId 未过滤）（🔴 Critical）

- **问题描述**：数据库查询未包含 `tenantId` 过滤条件，导致 A 租户（公司）的用户可能访问到 B 租户的数据。
- **典型反例**：
  ```ts
  const orders = await db.select().from(ordersTable); // 🚨 全表查询，跨租户泄露
  ```
- **正确写法**：
  ```ts
  // tenantId 来自 auth() 会话，绝不能由客户端传入
  const session = await auth();
  const tenantId = session!.user.tenantId;
  const orders = await db.select().from(ordersTable).where(eq(ordersTable.tenantId, tenantId)); // 强制过滤
  ```
- **快速检测**：`grep -rn "db\.select\(\)" src/features --include="*.ts" | grep -v "tenantId"`
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：finance (2026-03-10), orders (2026-03-10), customers (2026-03-10)

## D4 - 数据库与性能类

### D4-001 N+1 查询问题（🟠 High）

- **问题描述**：在循环中执行数据库查询，导致 N 条记录触发 N+1 次数据库请求，随着数据量增长性能急剧下降。
- **典型反例**：
  ```ts
  const orders = await db.select().from(ordersTable);
  for (const order of orders) {
    order.customer = await db.select()... // 每条订单查一次客户
  }
  ```
- **正确写法**：使用 `db.query.orders.findMany({ with: { customer: true } })` Drizzle 关联查询
- **快速检测**：在代码中搜索 `for...of` 或 `.map` 循环内部是否有 `await db.` 调用
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：-

---

### D4-002 无限制查询未设 LIMIT（🟠 High）

- **问题描述**：查询数据库时没有设置 `limit()`，当数据量大时可能返回成千上万条记录，导致内存溢出和页面崩溃。
- **典型反例**：
  ```ts
  const allCustomers = await db
    .select()
    .from(customersTable)
    .where(eq(customersTable.tenantId, tenantId)); // 🚨 可能返回 10000+ 条
  ```
- **正确写法**：必须添加 `.limit(pageSize).offset(offset)` 分页控制
- **快速检测**：`grep -rn "db\.select\(\)" src/features --include="*.ts" | grep -v "\.limit("`
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：finance (2026-03-10), quotes (2026-03-10), leads (2026-03-10)

## D5 - UI/UX 类

### D5-001 三态处理缺失（Loading/Empty/Error）（🟡 Medium）

- **问题描述**：组件只有"正常数据展示"状态，缺少加载中骨架屏（Loading）、空状态提示（Empty）、错误状态（Error）的处理，导致用户体验断层。
- **典型反例**：返回数据列表，但当 `data.length === 0` 时页面一片空白

  - Error → `error.tsx` 错误边界 + retry 按钮
- **快速检测**：检查 `page.tsx` 同级目录是否有 `loading.tsx` 和 `error.tsx`
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：-

### D5-002 手机浏览器响应式适配缺失（🟠 High）

- **问题描述**：PC Web 页面在手机浏览器（375px 视口）下出现内容溢出、表格无法横滚、弹窗超出屏幕、按钮触摸区域过小等问题，影响工人/经理在手机上的业务操作体验。
- **常见反例**：

  ```tsx
  {
    /* ❌ 表格裸放，无横向滚动容器 */
  }
  <Table>...</Table>;

  {
    /* ❌ 弹窗宽度固定，小屏溢出 */
  }
  <DialogContent className="w-[600px]">...</DialogContent>;

  {
    /* ❌ 图标按钮触摸区域 16×16px，手指难以点击 */
  }
  <Button variant="ghost" size="icon">
    <Icon className="h-4 w-4" />
  </Button>;
  ```

- **正确写法**：

  ```tsx
  {
    /* ✅ 表格外层加滚动容器 */
  }
  <div className="w-full overflow-x-auto">
    <Table>...</Table>
  </div>;

  {
    /* ✅ 弹窗自适应屏幕宽度 */
  }
  <DialogContent className="w-full max-w-lg sm:max-w-[600px]">...</DialogContent>;

  {
    /* ✅ 确保触摸最小目标 44×44px */
  }
  <Button variant="ghost" size="icon" className="h-11 w-11">
    <Icon />
  </Button>;

  {
    /* ✅ 金额输入框触发数字键盘 */
  }
  <Input type="text" inputMode="decimal" />;
  ```

- **快速检测**：Chrome DevTools → Toggle device toolbar → iPhone 375px 视口，检查页面是否有水平滚动条（`document.documentElement.scrollWidth > window.innerWidth`）
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：finance (2026-03-10)

## D8 - 可运维性类

### D8-001 写操作未记录审计日志（🟠 High）

- **问题描述**：创建、更新、删除等写操作没有调用 `AuditService.log()`，无法追溯"谁在什么时间做了什么操作"，出现数据异常时无法排查。
- **典型反例**：Server Action 中执行 `db.insert/update/delete` 后没有 `AuditService.log(...)` 调用
- **正确写法**：每个写操作后追加（**AuditService.log 第一个参数必须是 db实例**）：
  ```ts
  await AuditService.log(db, {
    tableName: 'orders',
    recordId: result.id,
    action: 'CREATE',
    userId: session.user.id,
    tenantId: session.user.tenantId,
    details: { data: result },
  });
  ```
- **快速检测**：对比 `db.insert|update|delete` 调用次数与 `AuditService.log` 调用次数是否匹配
- **首次发现**：全局规范制定 (2026-03-10)
- **复现记录**：quotes (2026-03-10)

### D4-003 乐观锁覆盖不完整（🟠 High）

- **问题描述**：同一模块内，部分写操作使用了乐观锁（`WHERE version = ?`），但另一部分同类操作遗漏乐观锁，形成安全不一致。并发时遗漏锁的操作可导致数据覆盖或金额双扣。
- **典型反例**：

  ```ts
  // ✅ 供应商付款单有乐观锁
  .where(and(eq(table.id, id), eq(table.version, version)))

  // 🚨 劳务结算单更新无乐观锁（同模块）
  .where(and(eq(table.id, id), eq(table.tenantId, tenantId)))
  ```

- **正确写法**：同模块内所有涉及金额或状态变更的 UPDATE 必须统一添加 `eq(table.version, currentVersion)` 条件
- **快速检测**：搜索同一模块 action 文件中的 `.update()` 调用，对比是否所有 WHERE 子句均含 `version` 字段
- **首次发现**：finance (2026-03-10)
- **复现记录**：-

### D8-002 遗留 console.log 未迁移至统一 logger（🟡 Medium）

- **问题描述**：部分 Server Action 或 Service 文件中使用了原生 `console.log/console.error`，而非项目统一的 `logger`（集成于结构化日志系统）。这导致日志无法被日志聚合平台采集、无法携带 traceId/tenantId 上下文。

### D3-006 TOCTOU 漏洞：查询加 tenantId、更新不加（🔴 Critical）

- **问题描述**：在"Check-Then-Act"模式下，前置查询（`findFirst`）使用了 `tenantId` 隔离，但实际的数据库 UPDATE 语句中 WHERE 子句只用 `id` 过滤。这形成“検查-使用”时间窗口（TOCTOU）：攻击者在成功通过权限查询后、实际执行更新前，替换记录属于另一租户，导致跨租户数据被第三方修改。
- **典型反例**：
  ```ts
  // ✅ 查询时有 tenantId
  const notice = await db.query.liabilityNotices.findFirst({
    where: and(eq(liabilityNotices.id, id), eq(liabilityNotices.tenantId, tenantId)),
  });
  // ❌ 更新时无 tenantId！TOCTOU 窗口
  await db.update(liabilityNotices).set({...}).where(eq(liabilityNotices.id, id));
  ```
- **正确写法**：UPDATE WHERE 子句必须加上 `and(eq(table.id, id), eq(table.tenantId, tenantId))`
- **快速检测**：搜索所有 `.update(...)` 语句，检查其 WHERE 是否仅包含主键而无 tenantId
- **首次发现**：after-sales (2026-03-10)
- **复现记录**：settings (2026-03-10) | approval (2026-03-10) | service (2026-03-10) | products (2026-03-10) | supply-chain (2026-03-10) | showroom (2026-03-10) | sales (2026-03-10) | ai-rendering (2026-03-10)

### D2-004 静默 Fallback 降级掉盖错误（🟡 Medium）

- **问题描述**：数据库或远程查询失败时，代码自动降级到更简单的备用查询并继续返回直至前端。这使得前端静默收到残缺数据（如缺少关联字段）且监控系统漏报错，难以排查根因。
- **典型反例**：
  ```ts
  const data = await db.query.customers.findMany({ with: { assignedSales: true }, ... })
    .catch(() =>
      // 🚨 关联失败，静默降级为无关联查询，前端收到空 assignedSalesName
      db.query.customers.findMany({ ... })
    );
  ```
- **正确写法**：直接抛出异常而非降级，向监控和页面错误边界暴露错误；若确实需要降级逻辑，必须先记录一条包含完整堆栈的 error 日志
- **快速检测**：搜索代码中的 `.catch(() =>` 模式，检查是否采用降级返回策略而非抛出
- **首次发现**：customers (2026-03-10)
- **复现记录**：-

### D3-005 密码重置前未清理旧 Token（Token 泛洪攻击）（🔴 Critical）

- **问题描述**：用户发起密码重置请求时，新 Token 写入 `verificationCodes` 前未清理该用户已有的同类型旧 Token。攻击者可无限次请求，导致数据库中堆积大量无效 Token，形成存储泛洪；若邮件服务为计费型，同时构成资金耗尽攻击。
- **典型反例**：
  ```ts
  // 🚨 直接 insert 新 token，不清理旧的
  await db.insert(verificationCodes).values({ userId, token, type: 'PASSWORD_RESET', ... });
  ```
- **正确写法**：
  ```ts
  // ✅ insert 前先删除同用户同类型的未使用 token
  await db.delete(verificationCodes).where(
    and(
      eq(verificationCodes.userId, user.id),
      eq(verificationCodes.type, 'PASSWORD_RESET'),
      eq(verificationCodes.used, false)
    )
  );
  await db.insert(verificationCodes).values({ ... });
  ```
- **快速检测**：找所有 `insert(verificationCodes)` 调用，检查其前方是否有对应的 `delete(verificationCodes)` 清理逻辑
- **首次发现**：auth (2026-03-10)
- **复现记录**：-

### C1-001 Props 接口含 `any` 类型（🟠 High）

- **问题描述**：组件的 Props 接口使用了 `any` 类型，导致类型保护失效，调用方传入错误类型不会触发编译错误。
- **典型反例**：
  ```tsx

    data: any; // 🚨 失去类型保护
    onSelect: (val: any) => void;
  }
  ```
- **正确写法**：
  ```tsx
  interface Props {
    data: Record<string, string>; // ✅ 精确类型
    onSelect: (val: string) => void;
  }
  ```
- **快速检测**：`grep -rn ": any" src/shared/ui src/widgets --include="*.tsx"`

### C2-001 渲染函数内内联创建对象/函数（🟡 Medium）

- **问题描述**：在组件渲染函数体内直接写 `onClick={() => ...}`、`style={{ color: 'red' }}`，每次渲染都会创建新引用，导致子组件无效重渲染（尤其当子组件用了 `React.memo`）。
- **典型反例**：
  ```tsx
  return <Button onClick={() => handleDelete(id)} />; // 🚨 每次渲染新函数
  ```
- **正确写法**：
  ```tsx
  const handleClick = useCallback(() => handleDelete(id), [id]); // ✅
  return <Button onClick={handleClick} />;
  ```
- **快速检测**：在组件文件中搜索 `onClick={() =>` 的模式，检查是否在性能敏感路径上
- **首次发现**：组件审计体系建立 (2026-03-11)
- **复现记录**：-

### C3-001 敏感操作缺少二次确认（🔴 Critical）

- **问题描述**：删除数据、清空记录、取消订单等不可逆操作，直接绑定到按钮点击事件，没有通过 `action-confirm-dialog` 弹出确认框，用户误触无法撤回。
- **典型反例**：
  ```tsx
  <Button onClick={() => deleteRecord(id)}>删除</Button> // 🚨 误触即删
  ```
- **正确写法**：
  ```tsx
  <ActionConfirmDialog
    title="确认删除？"
    description="此操作不可撤销"
    onConfirm={() => deleteRecord(id)}
  >
    <Button variant="destructive">删除</Button>
  </ActionConfirmDialog>
  ```
- **快速检测**：搜索组件中的「删除」「清空」「取消」「作废」按钮，检查是否被 `ActionConfirmDialog` 包裹
- **首次发现**：组件审计体系建立 (2026-03-11)
- **复现记录**：-

### C3-002 表单双重提交防护缺失（🟠 High）

- **问题描述**：表单提交按钮在 Server Action 执行期间未禁用，用户可以快速多次点击，触发重复的网络请求和数据库写操作。
- **典型反例**：
  ```tsx
  <Button type="submit">提交</Button> // 🚨 提交期间仍可点击
  ```
- **正确写法**：
  ```tsx
  const [state, action, isPending] = useActionState(submitForm, null);
  <Button type="submit" disabled={isPending}>
    {isPending ? '提交中...' : '提交'}
  </Button>;
  ```
- **快速检测**：搜索表单提交按钮，检查是否使用 `isPending`/`disabled` 防止双重提交
- **首次发现**：组件审计体系建立 (2026-03-11)
- **复现记录**：-

### C4-001 业务输入组件缺少错误态展示（🟠 High）

- **问题描述**：输入组件（如银行账号、货币、地址）只有正常渲染状态，当表单校验失败后没有视觉反馈（红色边框、错误文字提示），用户不知道哪里填错了。
- **典型反例**：
  ```tsx
  <Input value={value} onChange={...} /> // 🚨 无 error 状态
  ```
- **正确写法**：
  ```tsx
  <Input
    value={value}
    onChange={...}
    error={errors.bankAccount} // ✅ 传入错误信息
    aria-invalid={!!errors.bankAccount}
    aria-describedby="bank-error"
  />
  {errors.bankAccount && <p id="bank-error" className="text-destructive text-sm">{errors.bankAccount}</p>}
  ```
- **快速检测**：检查自定义输入组件的 Props 是否包含 `error` 字段，以及组件内部是否渲染 error 样式
- **首次发现**：组件审计体系建立 (2026-03-11)
- **复现记录**：-

### C5-001 图标按钮缺少 aria-label（🟡 Medium）

- **问题描述**：纯图标的操作按钮（铃铛、搜索、关闭、编辑）没有文字内容，屏幕阅读器无法告知视障用户该按钮的功能。
- **典型反例**：
  ```tsx
  <Button variant="ghost" size="icon">
    <Bell className="h-5 w-5" /> {/* 🚨 屏幕阅读器念出：「按钮」 */}
  </Button>
  ```
- **正确写法**：
  ```tsx
  <Button variant="ghost" size="icon" aria-label="查看通知">
    <Bell className="h-5 w-5" aria-hidden="true" />
  </Button>
  ```
- **快速检测**：`grep -n "size=\"icon\"" src/widgets src/shared/ui --include="*.tsx" -r | grep -v "aria-label"`
- **首次发现**：组件审计体系建立 (2026-03-11)
