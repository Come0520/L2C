# L2C SaaS 套餐与商业化计费体系设计 (Billing Tiers & Monetization)

## 1. 核心架构：固定层级 + 动态额度开关组合策略 (Hybrid Tier Model)

本方案采用 **选项 A（固定 Base、Pro、Enterprise 三个模板）** 作为系统对外展示与鉴权的基础锚点，但底层核心能力通过**数字限额 (Quantitative Limits)** 和 **布尔开关 (Feature Toggles)** 进行精细化配置。此设计在保证系统判断逻辑简单清晰的同时，兼顾了后期运营做活动或出组合包的灵活性。

### 1.1 套餐层级定义

- **Base (基础版)**: 永久免费。定位为生产力工具，满足入门级微型门店（1-3 人）的开单、展示与基本运作需求，但通过资源数量与长期存储成本做强制上限，锁定管理与扩展能力。
- **Pro (高级定制版)**: 定价 2999/月（暂定，可按需修改）。定位为成熟团队必备工具，提供“防内鬼审计”、“不限量建单”、“数据资产导出”等核心**管控+增效**特权。
- **Enterprise (私有化/旗舰版)**: 按需定制，针对有 API 对接 ERP、多门店跨区管理需求的头部集团。

## 2. 套餐差异化限额与功能控制表 (Limits & Toggles)

系统将提供给平台管理员一个配置界面，针对这三个层级调节以下参数。

### 2.1 强管控资源数字限额 (Numbers)

通过“业务体量”倒逼有实力的公司进行升级：

| 限额变量 | 含义说明 | Base (基础版) | Pro (专业版) | Enterprise (旗舰版) | 杠杆原理 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `maxUsers` | 最大员工支持数 | 3 人 | 15 人 | 无限制 (`-1`) | 拦截超过 3 人的团队，因为他们通常有利润空间购买专业版 |
| `maxCustomers` | 最大客户档案容量 | 500 个 | 无限制 | 无限制 | 触达 500 个客户意味着已经是稳定运营门店，需要付费突破瓶颈 |
| `maxQuotesPerMonth` | 每月最大报价单生成数 | 50 单 | 无限制 | 无限制 | 单量超过 50 的门店营收较好，应承担 SaaS 工具费用 |
| `maxShowroomProducts` | 云展厅上架商品数 | 100 个 | 500 个 | 无限制 | 限制重度使用的选品库，避免白嫖高昂 OSS 流量费用 |
| `maxStorageGb` | OSS 设计图纸云端容量 | 1 GB | 10 GB | 无限制 | 超出空间即提示升级，锁定历史设计资产 |

> **软上限超越设计 (Soft Limit) 建议**：
> 当用户使用报价单满 50 单时，系统**不拦截创建动作**，而是将超额报价单标记为 `[超额封锁]`，仅自己可见，**无法对外分享或收取订金**。这种策略能在不流失销售数据记账需求的前提下，极大概率促成充值。

### 2.2 高级功能与商业护城河开关 (Booleans)

通过隔离“管理与协作属性”做差异化：

| 开关变量 | 含义说明 (Upsell 亮点) | Base | Pro | Enterprise |
| :--- | :--- | :--- | :--- | :--- |
| `dataExport` | 允许到处客户与订单 Excel | ❌ | ✅ | ✅ |
| `removeWatermark` | 抹除生成的落地页 L2C 官方水印 | ❌ | ✅ | ✅ |
| `brandCustomization` | 上传专属 Logo、报价单自定义模板 | ❌ | ✅ | ✅ |
| `auditTrailRetention`| 操作日志审计时长（防内鬼查账） | ❌(0天) | ✅(90天)| ✅(永久) |
| `fineGrainedRbac` | 允许自定义不同员工岗位的精细权限 | ❌(仅只读) | ✅ | ✅ |
| `multiLevelApproval` | 多级会签/流转审批流 | ❌(单层审批) | ✅ | ✅ |
| `multiStore` | 多门店数据隔离与连锁管理 | ❌ | ❌ | ✅ |
| `apiAccess` | 开放 Open API/Webhook (接 ERP 用) | ❌ | ❌ | ✅ |
| `coldStorageRule` | 休眠档案6个月后自动降速转入冷存储 | ✅(强制执行) | ❌(极速秒开)| ❌(极速秒开)|

## 3. 生成图片点数 (AI Credits) 架构设计

考虑到生成图片需要实时消耗底层显卡/API 算力成本，这里彻底打破“死板包月”，使用一套独立的**积分经济模型**。

### 3.1 点数发放模型组合

1. **终身试用池 (Trial Credits - 仅限 Base 版新客)**
   - 给每个新注册的 Base 租户发放 **20 个终身额度**。
   - 目的：让他无负担体验高质量出图。用完即止，不参与下个月刷新（即 Base 套餐的包月配额为 `0`）。
   - 让有刚需的白嫖用户花完这 20 个点后，面临“必须付费买图”或“升级 Pro”的抉择。

2. **核心包月池 (Monthly Pledged Credits - 针对付费版)**
   - **Pro 版租户**：每月 1 号系统自动为其重置发放到规定的额度（例如 `500 点/月`），月底清零。
   - **Enterprise 版**：配置为 `-1`（不限出图数额）。

3. **单购加油包设计 (Add-on Credits - 额外收益渠道)**
   - 当 Pro 版用户这个月跑单太多（比如做大工程发了大量图），500 点耗光了，但他不想买昂贵的 Enterprise 版。
   - 允许他们单独扫码充值购买：“**￥99 购买 200 点永久额度加油包**”。
   - **扣费优先级**：有加油包时，优先扣**包月池额度**，包月池扣穿后再扣**加油包额外额度**（因为加油包永不过期，保证用户不吃亏）。

### 3.2 消耗与记账规则

- **积分消耗梯度**：根据算力负担调整标价
  - 普清基础图/简易重绘：扣除 1 点。
  - 4K 高清渲染/全景图/视频漫游生成：扣除 3 点。
- **补偿机制与幂等设计**：
  - 调用生图队列时，在 DB 采取 `Credits Reserve`（预扣减）动作。
  - 若因为 SD 模型崩溃/API 超时导致未能出图成功，必须立刻回滚退回扣除点数（`Refund Credits`），防止产生客诉。

## 4. 特殊增值场景 (Add-ons 扩展设计)

如果将来出现“我只想加 5 个员工，不想升级整个版本”这种高频散单需求，我们的后端结构将使用以下混合鉴权策略：

- 租户在数据库中的核心判断标准不再只是读取 `PLAN_LIMITS['base'].maxUsers`。
- 我们设计如下函数 `getPlanLimit(tenantId, 'users')`，伪代码处理逻辑为：
  ```ts
  const baseLimit = PLAN_LIMITS[tenant.planType].maxUsers;
  // 检查租户自身是否购买并堆叠了额外的增值包
  const addonLimit = tenant.purchasedAddons.usersLimitBonus || 0;
  return baseLimit + addonLimit;
  ```
这种做法完美接纳了所有的“特批”、“打折定制”和“流量主零碎加购”商业逻辑，不会写废核心代码。

## 5. 祖父条款与套餐快照机制 (Grandfathering & Snapshots)

为了保障系统“后台任意调配套餐规则”的能力，且**绝对不影响**已付款在履约期内的老客户，方案必须引入快照隔离机制。

### 5.1 废弃硬编码套餐读取
- 不再把当前的 `Pro` 版配置写死为全站通用的硬限制。
- 建立 `plan_definitions` (定价计划定义库)，采用版本化的概念生成方案包。例如 `pro_2026_q1` 和 `pro_2026_q2`。
- 当运营在后台调降 `pro_2026_q2` 的赠送额度并设定其为“当前在售激活 (`is_active: true`)”时，旧版的 `pro_2026_q1` 则自动下架不再允许新购。

### 5.2 订阅快照绑定
- 客户在 3 月份交钱购买 `Pro` 包年套餐时，在 `subscriptions` 表中创建为期 1 年的绑定关系，严格关联 `plan_code: pro_2026_q1`，此记录锁定 1 年。
- **运行表现**：当 6 月份系统读取这个客户的权限时，系统拉取的是他绑定的那个已经下架的 Q1 配额数据。老客户的承诺履约完全不受最新活动策略影响（即祖父条款）。直到次年 3 月他重签续约，再强制迁移至最符合当时的 Q2 或最新方案。

## 6. 商业化配置管理可视化平台 (Monetization Admin Panel)

平台后台必须具备直接干预系统计费资源的能力，无需修改代码发布：

1. **套餐 SKU 拖拽工厂**
   - 展现不同时期、不同变体版本的定价计划卡片。
   - 上市/下架操作，或者新建营销节点专用套餐（支持人数、功能开关的表单调节）。
2. **单租户超管面板 (Tenant Override Panel)**
   - 全局透视单店用量。
   - 在遇到大客户非标定制需求时，提供快捷按钮弹窗“下发私下达成协议的增配资源（Add-ons）”，如送 5个名额，送三个月去水印特权。
3. **白嫖党与黑产监控看板**
   - 过滤列出长据免费版且基础配额几乎爆满的头部租户。
   - 高频点数消耗异动检测。提供一键拨号跟进或强制阻断账户的商业武器。
